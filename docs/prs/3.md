# Issue #3: Phase 2 -- Render namespace

Branch: `3-render-namespace`

## Summary

Four pure functions in the Render namespace that convert data to HTML fragments. No HTTP, no Stark, no side effects. These are the building blocks for all server responses in later phases.

## Changes

### New files

| File | Description |
|------|-------------|
| `APLSource/Render/Escape.aplf` | HTML entity escaping. Replaces `&`, `<`, `>`, `"`, `'` with their entity equivalents. Ampersand processed first to prevent double-escaping. |
| `APLSource/Render/RespondHTML.aplf` | Dyadic helper. Sets content-type to `text/html; charset=utf-8` on the request, returns the HTML string. |
| `APLSource/Render/TodoItem.aplf` | Renders a single todo namespace as an `<li>` fragment with htmx attributes. Title is escaped. Completed items get `class="completed"`. |
| `APLSource/Render/TodoList.aplf` | Renders a vector of todos by concatenating `TodoItem` for each. Returns empty string for empty input. |

## Design decisions

1. **Escape uses vectorised replacement.** Each character in the input is enclosed, then masked positions are replaced with entity strings, and the result is enlisted. This avoids character-by-character looping while keeping the code readable. The ordering (ampersand first) is guaranteed by the `:For` loop over the character/entity pairs.

2. **TodoItem generates a flat character vector.** No line breaks or indentation in the output. This is deliberate: htmx swaps the fragment directly into the DOM, and whitespace between elements can cause layout issues. The architecture document shows indented HTML for readability, but the actual output is a single line.

3. **TodoList uses `TodoItem¨` (each).** Simple and direct. The `∊` (enlist) flattens the resulting vector of character vectors into a single string.

4. **RespondHTML is minimal.** It exists solely as a safety net to ensure content-type is always set. Two lines of code.
