# Architecture Review: Todo Application

## Summary verdict

The plan is a competent sketch of an htmx+APL todo application, but it contains one show-stopping error (HTMLInterface does not work in REST paradigm), one high-severity misunderstanding of request payload format, and several medium-severity gaps in error handling, concurrency, and testing. The proposed `GET /` route for serving the shell page will not work as described; the plan needs a different strategy for serving static files. The HTML-returning approach via `SetContentType` is technically sound but the plan omits important detail about what Jarvis actually does with the response payload in that case.

## Issues found

1. **CRITICAL -- HTMLInterface is incompatible with REST paradigm.** The plan states (line 61): "Jarvis's `HTMLInterface` property will be pointed at the `static/` directory to serve `index.html`, `style.css`, and any other static assets." This will not work. Jarvis explicitly rejects `HTMLInterface←1` when `Paradigm` is not `'JSON'` (Jarvis.aplc line 114: `Log'HTML interface is only available using JSON paradigm'`). Stark forces `Paradigm←'REST'` (Stark.aplc line 325). The HTML file-serving code path (`handleHtml:` at Jarvis.aplc line 1096) lives inside `HandleJSONRequest`, not `HandleRESTRequest`. There is no static file serving capability in REST mode whatsoever. The entire static-file strategy must be rethought.

2. **CRITICAL -- `GET /` route will not serve a file.** The plan proposes `GET /` as a route that "serves the shell page" (line 53). But Stark handlers return APL values that Jarvis serialises. Even if a handler reads `index.html` from disk and returns the content as a character vector, Jarvis will attempt to JSON-serialise it unless the content-type is changed first (see issue 4 below). More fundamentally, this conflates two concerns: the plan says HTMLInterface handles static files for "paths that do not match a registered route" (line 61), but `GET /` *is* a registered route. This creates a contradictory design: is `/` handled by Stark or by HTMLInterface? In reality, since HTMLInterface will not work at all (issue 1), this is moot, but the plan is internally inconsistent regardless.

3. **HIGH -- htmx sends `application/x-www-form-urlencoded`, not JSON.** The plan states (line 110): "read `req.Payload.title`" for `CreateTodo`. By default, an HTML `<form>` with `hx-post` sends data as `application/x-www-form-urlencoded`, not `application/json`. In REST mode, Jarvis does parse form-urlencoded data (Jarvis.aplc line 1284-1285), but the resulting `Payload` is a namespace where each field value is a *nested vector* (because Jarvis accumulates values to handle repeated field names -- see `ParseUrlencodedForm` at line 1257-1267, where each field becomes a vector via `,←`). The plan assumes `req.Payload.title` will be a simple string, but it will be an enclosed vector containing a string. This will cause runtime errors or incorrect behaviour unless unwrapped. The plan should either: (a) document that form fields need unwrapping (e.g. `title←⊃req.Payload.title`), or (b) configure the htmx form to send JSON via `hx-ext="json-enc"`, which requires loading an additional htmx extension.

4. **HIGH -- JSON serialisation bypass needs precise sequencing.** The plan's code example (lines 126-129) shows `req.SetContentType 'text/html; charset=utf-8'` called inside the handler. This is the correct mechanism -- Jarvis.aplc line 1321 checks the content-type header and only calls `ToJSON` if it matches `application/json`. However, the plan does not mention that Stark's `_Dispatch` (Stark.aplc line 189) captures the handler's *return value* and passes it back to Jarvis as the response payload. Jarvis then sets `ns.Req.Response.Payload←resp` (line 1308). At line 1317-1318, if no content-type has been set on the response headers, Jarvis applies `DefaultContentType` (which is `application/json; charset=utf-8`). The `SetContentType` call inside the handler *does* set the header before this check, so it should work -- but the plan presents this as straightforward when the chain is actually fragile. If a handler forgets the `SetContentType` call, the HTML string will be JSON-encoded (wrapped in quotes, special characters escaped), producing garbled output. There is no safety net. The plan should recommend a wrapper function or middleware pattern to enforce this.

5. **HIGH -- `req.SetContentType` exists but the plan references it incorrectly.** The method exists at Jarvis.aplc line 1741 as `SetContentType contentType` on the Request class. The plan's syntax `req.SetContentType 'text/html; charset=utf-8'` is correct APL for calling a monadic method. This specific claim checks out. However, the Stark router documentation (stark-router.md line 157) states: "A handler is a monadic function that takes `req` and returns a namespace (which Jarvis serializes to JSON)." The plan proposes returning a *character vector* instead of a namespace. This works in practice (Jarvis's `ToJSON` is only called when content-type is `application/json`, and `Respond` at line 1404 sends `res.Payload` as-is), but it contradicts the documented handler contract and could break if Stark or Jarvis adds response validation in future.

6. **MEDIUM -- No strategy for serving static CSS/JS files.** Since HTMLInterface does not work in REST mode (issue 1), there is no mechanism to serve `style.css` or any other static asset from the `static/` directory. The plan must either: (a) add explicit Stark routes for each static file (ugly, fragile), (b) use a separate static file server or reverse proxy (nginx, Caddy), (c) inline all CSS in `index.html`, or (d) create a catch-all Stark handler that reads files from disk. Option (d) would require understanding that Stark does not support wildcard/catch-all routes (no evidence of `*` or `**` patterns in the routing code). This is a significant gap.

7. **MEDIUM -- Concurrent access to the `todos` vector is unprotected.** The plan acknowledges in-memory storage (line 45) but does not mention thread safety. Jarvis's default `ThreadMode` spawns each request in a new thread (Jarvis default, or `ThreadMode←1`). Multiple concurrent requests could corrupt the `todos` vector -- e.g., two simultaneous `POST /todos` could use the same `nextId` value, or a `DELETE` concurrent with a `ListTodos` could produce an index error. The plan should either: (a) use `ThreadMode←0` (single-threaded, simplest for a demo), (b) use `:Hold` tokens around data mutations, or (c) document this as a known limitation.

8. **MEDIUM -- Path parameter `id` is a string, not an integer.** The plan's data model uses integer `id` fields (line 40), but path parameters are always strings (documented in stark-router.md line 124-128). The plan's handler descriptions (lines 111-112) say "find by `req.PathParams.id`" and "flip `completed`" without mentioning the string-to-integer conversion needed. The example app (ExampleApp.aplc line 99) shows the correct pattern: `id←⊃2⊃⎕VFI req.PathParams.id`. The plan should specify this conversion and handle the case where `id` is not a valid integer (returning 400, not crashing).

9. **MEDIUM -- No mention of htmx-specific request headers.** htmx sends several custom headers: `HX-Request: true`, `HX-Target`, `HX-Trigger`, `HX-Current-URL`, and others. The plan does not mention these at all. While not strictly required for basic operation, they are important for: (a) distinguishing htmx requests from full-page loads (e.g., a user navigating directly to `/todos` in the browser should get a full page, not a fragment), (b) returning appropriate responses when `HX-Target` differs from what was expected, and (c) potentially using `HX-Trigger` response headers to trigger client-side events (e.g., updating a todo count). The plan should at minimum document that `HX-Request` is not checked.

10. **MEDIUM -- ID reuse after deletion is not addressed.** The plan uses an auto-incrementing `nextId` counter (line 99), which means IDs are never reused. This is fine. However, the plan does not address what happens when a `PUT /todos/{id}` or `DELETE /todos/{id}` is called with an ID that does not exist. The handler descriptions (lines 111-112) do not mention 404 responses for missing IDs. The ExampleApp handlers do handle this (e.g., ExampleApp.aplc lines 101-103), so the pattern exists, but the plan omits it.

11. **MEDIUM -- `dyalogscript` source loading mechanism is underspecified.** The plan's entry point (lines 135-140) says it "loads Stark and Jarvis into the workspace" and "loads the application source". The ExampleApp's `Run.apls` (line 6-7) uses `⎕SE.Link.Import` for this. The plan does not specify whether `Link.Import` or `2⎕FIX` or some other mechanism is used. Importantly, `Link.Import` requires the files to be in specific formats (`.aplc` for classes, `.apln` for namespaces, `.aplf` for functions). The proposed `Render.apln` namespace would need its functions defined as individual `.aplf` files inside a `Render/` directory, or as a single `.apln` file with all functions inline. The plan's layout (line 25) suggests a single `Render.apln` file, which would need to contain all function definitions using the `:Namespace ... :EndNamespace` wrapper syntax. This is feasible but the plan should be explicit.

12. **LOW -- `GET /todos` route purpose overlaps with `hx-trigger="load"`.** The front-end design (line 71) has the `<ul>` trigger a `GET /todos` on page load. This is correct for htmx. But the plan also has `GET /` serve the shell page, which implies the user first loads `/`, then immediately fires `GET /todos`. If the `GET /` handler reads and returns the `index.html` file, this is fine. But if someone navigates directly to `/todos` in the browser, they get a bare HTML fragment, not a page. This is a standard htmx trade-off but the plan does not acknowledge it or suggest how to handle it (e.g., checking `HX-Request` header).

13. **LOW -- Testing strategy lacks edge case coverage.** The testing section (lines 146-153) covers the happy path (CRUD cycle, correct status codes, HTML fragments) but does not mention: (a) invalid input (empty title, missing fields, non-numeric ID), (b) operations on non-existent IDs, (c) XSS prevention (what if a todo title contains `<script>`?), (d) concurrent request handling, (e) large payloads or very long todo titles. The integration tests mention `HttpCommand` but do not mention testing response headers beyond content-type (e.g., verifying no `Content-Type: application/json` leaks through).

14. **LOW -- No XSS protection mentioned.** The plan renders user-supplied text (`title`) directly into HTML fragments (line 79: `Buy milk` inside a `<span>`). If a user submits a title like `<script>alert('xss')</script>`, the `Render.TodoItem` function must HTML-escape it. The plan does not mention HTML escaping anywhere. This is a security concern even for a demo application.

15. **LOW -- OpenAPI spec will describe JSON responses, not HTML.** Stark auto-registers `/openapi.json` (Stark.aplc line 40) and generates an OpenAPI spec assuming `application/json` responses (Stark.aplc line 221: `ctKey←_Mangle'application/json'`). The generated spec will be incorrect for this application since all endpoints return `text/html`. This is cosmetic for a demo but worth noting.

## Recommendations

1. **Replace the HTMLInterface strategy entirely.** Since Stark uses REST paradigm and HTMLInterface only works in JSON paradigm, static file serving must be handled differently. The simplest approach for a demo: (a) create a Stark route `GET /` whose handler reads `static/index.html` from disk and returns it as a character vector after calling `req.SetContentType`, (b) create a `GET /style.css` route that does the same for CSS, (c) alternatively, inline all CSS in the HTML file to avoid the problem. For a production approach, use a reverse proxy (nginx/Caddy) for static files.

2. **Account for form-urlencoded payloads.** Either: (a) add `hx-ext="json-enc"` to the htmx form and load the `json-enc` extension from CDN, ensuring `req.Payload` is a standard JSON-parsed namespace, or (b) document that `req.Payload.title` values from form-urlencoded submissions are enclosed vectors and unwrap them in handlers with `⊃req.Payload.title`.

3. **Create a helper function for HTML responses.** Rather than requiring every handler to call `req.SetContentType`, create a utility (e.g., in `Render.apln`) that wraps the pattern:
   ```
   result←html RespondHTML req
     req.SetContentType 'text/html; charset=utf-8'
     result←html
   ```
   This reduces the risk of a handler forgetting the call and producing JSON-encoded HTML.

4. **Use `ThreadMode←0` for simplicity.** Since this is a demo with in-memory state, single-threaded operation avoids all concurrency issues. Document this choice explicitly.

5. **Add HTML escaping to the Render namespace.** At minimum, escape `&`, `<`, `>`, `"`, and `'` in user-supplied strings before embedding them in HTML. This is straightforward in APL with a simple replacement chain.

6. **Add error cases to handler specifications.** Each handler description should specify what happens for: missing/invalid ID (400 or 404), missing required fields (400), empty title (400). The ExampleApp already demonstrates this pattern.

7. **Specify the `⎕VFI` conversion for path parameter IDs.** Explicitly state that `req.PathParams.id` is a string and must be converted, and that invalid values should return 400.

8. **Add `HX-Request` header checking to `GET /` at minimum.** If someone navigates directly to `/todos`, they should either get a redirect to `/` or a full page, not a bare fragment.

## What the plan gets right

- The overall architecture (Stark on Jarvis, htmx for the front-end, in-memory storage) is appropriate for a demo application and correctly identifies the key technologies.
- The htmx fragment pattern (returning `<li>` elements, using `hx-swap="outerHTML"` and `hx-swap="delete"`) is idiomatic and well-designed.
- The separation of rendering into `Render.apln` is a good design decision that enables unit testing without HTTP.
- The data model (simple namespace vector with auto-incrementing ID) is suitable for the scope.
- The `SetContentType` mechanism for bypassing JSON serialisation is the correct approach -- the plan identified the right Jarvis API for this.
- The route registration syntax matches Stark's actual API (dyadic verb methods with path on left, handler name on right).
- Scoping boundaries are clearly stated, which prevents scope creep.
